## Resumen

La sección de Acciones y Herramientas permite configurar funcionalidades avanzadas que potencian las capacidades de tu agente conversacional durante y después de las llamadas. Incluye transferencias telefónicas con reglas múltiples, detección automática de contestadores, webhooks personalizados, protocolos de rellamada, el **Schedule Controller (Beta)** para horarios de atención, integración de calendario y comunicación multicanal con Email, SMS y WhatsApp, incluyendo la vinculación de tu propio dominio de email.

<Info>
Las funcionalidades configuradas aquí son clave para crear experiencias conversacionales fluidas y automatizaciones poderosas que integren con tus sistemas existentes, maximizando la eficiencia operacional y la satisfacción del cliente.
</Info>

## Transferencia de Llamadas

Configura cómo tu agente maneja las transferencias de llamada a agentes humanos, extensiones internas o sistemas telefónicos externos con plantillas multiidioma optimizadas.

<Info>
Meetzy ofrece dos versiones del sistema de transferencia: la versión clásica **v1** (destino único) y la nueva **Transfer v2 (Beta)**, que permite definir múltiples reglas de transferencia (`TransferRule`) con condiciones independientes. Puedes alternar entre ambas desde el interruptor "Transfer to multiple destinations (Beta)" en el panel de configuración; internamente esto se controla mediante la propiedad `transferConfigVersion` (`'v1' | 'v2'`), que por defecto es `'v1'` para no alterar el comportamiento de agentes ya existentes.
</Info>

### Tipos de Transferencia Disponibles

Meetzy soporta dos métodos principales de transferencia con configuración flexible:

<Tabs>
  <Tab title="Número de Teléfono">
    Transferir a números telefónicos tradicionales (móvil, fijo, internacional).
    
    **Formato requerido**: `+34600123456`
    
    **Casos de uso comunes**:
    - Escalar a agentes de soporte humano
    - Conectar con departamentos específicos
    - Transferir a supervisores o especialistas
    - Dirigir a líneas de emergencia
    - Routing geográfico por ubicación del cliente
  </Tab>
  <Tab title="SIP Trunk">
    Transferir a través del protocolo SIP para sistemas telefónicos empresariales.
    
    **Formato requerido**: `sip:extensión@gateway_ip:puerto`
    
    **Ejemplo**: `sip:100@78.136.102.246:5060`
    
    **Casos de uso comunes**:
    - Transferir a extensiones internas de PBX
    - Conectar con sistemas VoIP empresariales
    - Integrar con infraestructura telefónica existente
    - Enrutar a grupos de agentes específicos
    - Queue management inteligente
  </Tab>
</Tabs>

### Proceso de Configuración

<Steps>
  <Step title="Activar Transferencia">
    Habilita la opción de transferencia de llamadas usando el interruptor en la interfaz de usuario.
  </Step>
  <Step title="Elegir Versión del Sistema de Transferencia">
    Decide si usarás el modo clásico (`transferConfigVersion: 'v1'`, un único destino) o **Transfer v2 (Beta)** (`transferConfigVersion: 'v2'`), que permite configurar múltiples reglas con condiciones distintas.
  </Step>
  <Step title="Seleccionar Tipo de Transferencia">
    Elige entre "Número de Teléfono" o "SIP Trunk" según tu infraestructura de comunicaciones (esta elección se hace por cada regla en v2).
  </Step>
  <Step title="Configurar Destino">
    - Para números telefónicos: Introduce el número en formato E.164 (+país + número)
    - Para SIP: Especifica la URI completa con extensión, IP y puerto
  </Step>
  <Step title="Definir Plantilla de Transferencia">
    Selecciona una plantilla predefinida optimizada para tu idioma o crea una personalizada.
  </Step>
  <Step title="Validar Configuración">
    Prueba la configuración con llamadas de test para verificar conectividad.
  </Step>
</Steps>

### Transfer v2 (Beta): Reglas Múltiples de Transferencia

<Info>
**Transfer v2 (Beta)** es la evolución del sistema de transferencia clásico. En lugar de definir un único destino y una única plantilla de condiciones, permite crear **múltiples reglas independientes** (representadas internamente por la interfaz `TransferRule`), cada una con su propia condición de activación y su propio destino. Esto es ideal para agentes que deben enrutar llamadas a distintos departamentos, idiomas o niveles de prioridad según el motivo de la consulta.
</Info>

Para activar esta funcionalidad, utiliza el interruptor **"Transfer to multiple destinations (Beta)"** que aparece al habilitar la transferencia de llamadas. Al activarlo, la interfaz cambia de un formulario de destino único a una lista dinámica de reglas (`transferRules: TransferRule[]`).

#### Cómo Funciona una Regla de Transferencia

Cada regla (`TransferRule`) se compone de dos partes:

| Componente | Descripción | Ejemplo |
|-----------|-------------|---------|
| **Condición** | Frase en lenguaje natural que describe cuándo debe aplicarse la regla. La IA evalúa esta condición durante la conversación. | "el cliente pregunta por facturación o pagos" |
| **Destino** | Número de teléfono o URI SIP al que se transferirá la llamada cuando se cumpla la condición | `+34911234567` o `sip:200@gateway.empresa.com:5060` |

<Steps>
  <Step title="Activar Transfer v2">
    Activa el interruptor "Transfer to multiple destinations (Beta)" dentro de la sección de Call Transfer. Esto fija `transferConfigVersion` a `'v2'`.
  </Step>
  <Step title="Añadir Reglas">
    Haz clic en "Añadir Regla" para crear un nuevo elemento en el array `transferRules`. Puedes crear tantas reglas como necesites.
  </Step>
  <Step title="Definir la Condición">
    Escribe en el campo "When should this be used?" una descripción clara de la situación que debe activar esta regla (ej. "el cliente solicita cancelar su suscripción").
  </Step>
  <Step title="Elegir Tipo de Destino">
    Selecciona "Phone" o "SIP" para cada regla de forma independiente.
  </Step>
  <Step title="Configurar el Valor de Destino">
    Introduce el número en formato E.164 o la URI SIP completa según el tipo seleccionado.
  </Step>
  <Step title="Priorizar y Ordenar Reglas">
    Ordena las reglas de mayor a menor prioridad; la IA evaluará las condiciones en el orden establecido dentro del array `transferRules` y aplicará la primera que coincida.
  </Step>
  <Step title="Eliminar o Editar Reglas">
    Usa el botón "Remove" en cada tarjeta de regla para eliminarla, o edita directamente los campos para actualizarla.
  </Step>
</Steps>

#### Ejemplo de Configuración con Múltiples Reglas

```json
{
  "transferConfigVersion": "v2",
  "transferEnabled": true,
  "transferRules": [
    {
      "condition": "El cliente pregunta por facturación, pagos o cargos duplicados",
      "destination": {
        "type": "phone",
        "value": "+34911234567"
      }
    },
    {
      "condition": "El cliente solicita soporte técnico o reporta un problema con el producto",
      "destination": {
        "type": "sip",
        "value": "sip:200@gateway.empresa.com:5060"
      }
    },
    {
      "condition": "El cliente pide expresamente hablar con un supervisor o gerente",
      "destination": {
        "type": "phone",
        "value": "+34911234568"
      }
    },
    {
      "condition": "El cliente muestra alta frustración o el caso requiere atención urgente",
      "destination": {
        "type": "sip",
        "value": "sip:911@gateway.empresa.com:5060"
      }
    }
  ]
}
```

#### Transfer v1 vs Transfer v2 (Beta): Comparativa

| Característica | Transfer Clásico (v1) | Transfer v2 (Beta) |
|----------------|------------------------|---------------------|
| **Número de destinos** | Uno único | Múltiples, sin límite práctico (`TransferRule[]`) |
| **Condiciones** | Una plantilla general multiidioma | Una condición independiente por regla |
| **Enrutamiento por departamento** | No soportado nativamente | Soportado de forma nativa |
| **Priorización de reglas** | No aplica | Sí, orden configurable según posición en el array |
| **Complejidad de configuración** | Baja | Media (requiere definir cada regla) |
| **Casos de uso ideales** | Agentes con un único punto de escalado | Contact centers con múltiples departamentos o niveles de servicio |

<Warning>
Transfer v2 se encuentra en fase **Beta**. Es recomendable probar exhaustivamente cada regla con llamadas de test antes de desplegar en producción, especialmente cuando existen condiciones que podrían solaparse entre sí. En caso de solapamiento, la IA aplicará la primera regla que coincida según el orden configurado en `transferRules`.
</Warning>

#### Buenas Prácticas para Reglas de Transferencia

- **Condiciones mutuamente excluyentes**: Redacta las condiciones de forma que no se superpongan entre sí para evitar ambigüedad en el enrutamiento
- **Orden por especificidad**: Coloca las reglas más específicas primero y las más genéricas al final como fallback
- **Regla de respaldo (fallback)**: Considera añadir siempre una última regla genérica ("cualquier otra situación que requiera atención humana") apuntando a un destino general
- **Nombres descriptivos en la condición**: Usa lenguaje natural claro y verbos de acción ("el cliente solicita...", "el cliente pregunta por...") para maximizar la precisión de detección de la IA
- **Pruebas por regla**: Valida cada regla individualmente con llamadas de test simulando distintos escenarios de conversación
- **No mezclar versiones**: Si migras de v1 a v2, revisa que la plantilla de condición clásica quede reflejada como una o varias reglas equivalentes en `transferRules` antes de desactivar el modo clásico

### Condiciones Adicionales de Finalización

El agente finalizará llamadas automáticamente en estas situaciones optimizadas:

- **Despedida Natural**: El usuario se despide claramente y el agente completa protocolarmente su despedida
- **Objetivos Completados**: Se han alcanzado exitosamente todos los objetivos específicos de la conversación
- **Timeout Configurado**: Se supera el tiempo máximo establecido para la duración de la llamada (configurable por agente)
- **Detección de Silencio Prolongado**: Períodos extendidos sin respuesta audible del interlocutor (umbral ajustable)
- **Interrupción de Línea**: Llamada cortada inesperadamente por problemas técnicos o cuelgue del cliente
- **Transferencia Exitosa**: Después de completar satisfactoriamente la transferencia a un agente humano
- **Detección de Hostilidad**: Cuando se identifica agresividad sostenida o abuso verbal del interlocutor
- **Compliance Violation**: Si se detecta solicitud de opt-out o violación de políticas de comunicación

## Schedule Controller (Beta)

<Info>
**Schedule Controller (Beta)** es la nueva herramienta del editor avanzado (respaldada internamente por las interfaces `ScheduleController`, `ScheduleDayConfig` y `ScheduleHoliday`) que permite que tu agente conozca en tiempo real si la llamada se está produciendo dentro o fuera del horario comercial configurado, y actúe en consecuencia (informar al usuario, ofrecer alternativas, activar un recall, o transferir a otro destino según el horario).
</Info>

### ¿Qué Resuelve Esta Herramienta?

Antes de esta funcionalidad, gestionar el horario de atención requería lógica externa vía webhooks personalizados. Ahora, el Schedule Controller se integra de forma nativa en el editor avanzado como una herramienta más que la IA puede consultar durante la conversación, sin necesidad de configurar infraestructura adicional.

**Casos de uso principales:**

- Informar al cliente si está llamando fuera de horario y comunicar el próximo horario de apertura
- Evitar transferencias a agentes humanos cuando el departamento está cerrado
- Activar automáticamente el protocolo de Recall para reintentar la llamada en horario laboral
- Ofrecer un mensaje diferenciado según sea horario laboral, fin de semana o festivo
- Enrutar a un buzón de mensajes o a un flujo de auto-servicio fuera de horario

### Modelo de Datos: `ScheduleController`

El Schedule Controller se construye sobre tres piezas de configuración claramente separadas, igual que un experimento correctamente controlado necesita variables independientes bien definidas:

| Interfaz | Responsabilidad | Ejemplo de Contenido |
|----------|------------------|------------------------|
| **`ScheduleController`** | Objeto raíz: activación, zona horaria y comportamiento fuera de horario | `enabled`, `timezone`, `outOfHoursBehavior` |
| **`ScheduleDayConfig`** | Configura un día concreto de la semana (abierto/cerrado, hora de inicio y fin) | `{ open: "09:00", close: "18:00" }` o `{ closed: true }` |
| **`ScheduleHoliday`** | Excepción puntual que sobrescribe el horario habitual en una fecha específica | `{ date: "2025-12-25", closed: true, reason: "Navidad" }` |

### Proceso de Configuración

<Steps>
  <Step title="Activar la Herramienta">
    Habilita "Schedule Controller" desde el interruptor correspondiente en la sección de Acciones y Herramientas (`enabled: true`).
  </Step>
  <Step title="Definir Horario Semanal">
    Configura la hora de inicio y fin para cada día de la semana usando un objeto `ScheduleDayConfig` por día. Puedes marcar días completos como cerrados (ej. domingos) con `closed: true`.
  </Step>
  <Step title="Establecer Zona Horaria">
    Selecciona la zona horaria de referencia (`timezone`) para que los cálculos de horario sean precisos independientemente de dónde llame el cliente.
  </Step>
  <Step title="Añadir Excepciones y Festivos">
    Define fechas específicas (festivos nacionales, vacaciones de empresa, eventos especiales) como entradas `ScheduleHoliday` que sobrescriban el horario habitual.
  </Step>
  <Step title="Configurar el Prompt de Uso">
    Describe cuándo y cómo la IA debe consultar esta herramienta durante la conversación (ej. al inicio de la llamada, o cuando el cliente solicite ser transferido).
  </Step>
  <Step title="Definir Comportamiento Fuera de Horario">
    Establece la acción a ejecutar (`outOfHoursBehavior`) cuando la llamada ocurra fuera de horario: mensaje informativo, activación de Recall, o transferencia a un destino alternativo.
  </Step>
</Steps>

### Comportamientos Disponibles Fuera de Horario

| Comportamiento | Descripción | Caso de Uso Recomendado |
|----------------|-------------|--------------------------|
| **Solo Informar** | El agente comunica el horario y continúa la conversación con opciones limitadas | Consultas informativas generales |
| **Informar y Ofrecer Recall** | Informa del horario y activa automáticamente el protocolo de Recall para la próxima apertura | Ventas y soporte que requieren atención humana |
| **Transferencia Alternativa** | Redirige a un destino distinto (ej. línea de urgencias 24h) en vez del destino habitual | Servicios con guardia o soporte crítico |
| **Mensaje y Finalización** | Informa del horario y finaliza la llamada educadamente | Negocios sin necesidad de seguimiento posterior |

### Integración con Otras Herramientas

<Tabs>
  <Tab title="Con Transferencia">
    Cuando el Schedule Controller está activo, puedes condicionar las reglas de Transfer v2 (Beta) al horario comercial. Por ejemplo, definir una `TransferRule` que transfiera a un número de guardia únicamente si la consulta es urgente **y** ocurre fuera de horario, evitando transferencias a un departamento cerrado.
  </Tab>
  <Tab title="Con Recall">
    El Schedule Controller se integra de forma nativa con el Protocolo de Rellamada: si la llamada ocurre fuera de horario y el objetivo no puede completarse, el sistema puede programar automáticamente un recall dentro de la próxima ventana de horario comercial disponible, respetando la configuración de `scheduleController` del sistema de Recall.
  </Tab>
  <Tab title="Con Webhooks">
    Puedes usar el estado del Schedule Controller (abierto/cerrado) como parámetro estático o dinámico dentro de tus herramientas webhook personalizadas, permitiendo que sistemas externos reciban contexto sobre si la llamada se produjo dentro o fuera de horario.
  </Tab>
</Tabs>

<Warning>
El Schedule Controller se encuentra actualmente en fase **Beta**. Verifica siempre la zona horaria configurada (`timezone`) y realiza pruebas en distintos horarios (dentro, fuera, y en el límite exacto de apertura/cierre) antes de desplegar en producción, especialmente si tu negocio opera en múltiples zonas horarias o países.
</Warning>

### Buenas Prácticas para el Schedule Controller

- **Zona horaria explícita**: Configura siempre la zona horaria de tu negocio en `timezone`, no asumas la zona horaria del cliente que llama
- **Mensajes claros con horario completo**: Incluye el horario completo semanal en el `messageTemplate` fuera de horario para reducir llamadas repetidas
- **Excepciones actualizadas**: Mantén actualizada la lista `holidays` de festivos y fechas especiales, revisándola trimestralmente
- **Combinar con Recall**: Para negocios orientados a ventas, combina el Schedule Controller con Recall para no perder oportunidades comerciales fuera de horario
- **Pruebas en bordes horarios**: Testea el comportamiento justo en los minutos de apertura y cierre (`open`/`close`) para evitar falsos negativos, con el mismo rigor con el que se aíslan variables en un experimento controlado

## Herramientas Webhook Personalizadas

Crea integraciones poderosas con sistemas externos mediante webhooks que tu agente puede ejecutar durante las conversaciones en tiempo real, con configuración avanzada y monitoreo completo.

### Creando una Nueva Herramienta Webhook

<Steps>
  <Step title="Iniciar Creación">
    Haz clic en "Añadir Herramienta Webhook" para crear una nueva integración personalizada con sistemas externos.
  </Step>
  <Step title="Configuración Básica">
    - **Nombre de Herramienta**: Identificador único y descriptivo (ej. "consultar_inventario", "crear_lead_crm")
    - **Descripción Detallada**: Explica específicamente cuándo y por qué la IA debe usar esta herramienta
    - **Contexto de Uso**: Define escenarios precisos que deben activar la herramienta automáticamente
  </Step>
  <Step title="Configuración de Endpoint">
    - **URL de Destino**: Endpoint HTTP/HTTPS que procesará las peticiones
    - **Método HTTP**: Selecciona GET, POST, PUT, PATCH o DELETE según la operación requerida
    - **Encabezados**: Configura autenticación, content-type y metadatos requeridos
  </Step>
  <Step title="Definir Parámetros">
    Configure parámetros de query string y campos del cuerpo de petición con validaciones específicas.
  </Step>
  <Step title="Configuración Avanzada">
    Ajusta timeouts, comportamiento de interrupciones, modo de ejecución y feedback de audio personalizado.
  </Step>

### Configuración Detallada de Parámetros

Cada parámetro webhook se puede configurar con precisión técnica avanzada:

| Propiedad | Descripción | Opciones Disponibles | Ejemplos de Uso |
|----------|-------------|---------------------|-----------------|
| **Identificador** | Nombre del parámetro en la API | string único (snake_case recomendado) | customer_id, product_sku, appointment_date |
| **Tipo de Dato** | Formato y validación del valor | string, number, boolean, integer, array, object | string para emails, integer para IDs |
| **Campo Requerido** | Obligatoriedad para la ejecución | true (obligatorio) / false (opcional) | email required: true, notes optional: false |
| **Descripción para IA** | Contexto detallado que guía el uso | texto descriptivo y ejemplos de uso | "Email del cliente obtenido durante conversación" |
| **Fuente del Valor** | Origen de los datos | llm_prompt (IA genera) / static (valor fijo) | IA extrae email, valor fijo: "source": "phone_call" |
| **Valores Enum** | Restricción a opciones predefinidas | lista cerrada de valores permitidos | ["urgente", "normal", "bajo"] para prioridades |
| **Validación** | Reglas adicionales de formato | regex, rangos numéricos, longitud mín/máx | email regex, telefono 9-15 dígitos |

### Configuración Avanzada de Webhooks

#### Control de Interrupciones Durante Ejecución

<Info>
Controla si el usuario puede interrumpir al agente mientras espera la respuesta del webhook. Esta configuración es crucial para operaciones críticas que requieren completarse sin interrupciones, mejorando la fiabilidad de transacciones importantes.
</Info>

- **Deshabilitar Interrupciones**: El agente no puede ser interrumpido durante la ejecución (ideal para transacciones críticas)
- **Permitir Interrupciones**: El usuario puede interrumpir mientras se procesa (recomendado para consultas informativas rápidas)

#### Modos de Ejecución de Webhooks

| Modo | Comportamiento | Casos de Uso Ideales | Ventajas | Desventajas |
|------|---------------|---------------------|----------|-------------|
| **Inmediato** | Ejecuta durante la conversación | Consultas de datos, validaciones en tiempo real | Respuesta inmediata, UX fluida | Puede afectar fluidez si es lento |
| **Diferido** | Ejecuta después de finalizar llamada | Logs, notificaciones, actualizaciones CRM | No interrumpe conversación | Usuario no recibe confirmación directa |

#### Configuración de Feedback de Audio

- **Sonido de Herramienta**: Reproduce efecto sonoro de "tecleado" durante ejecución para proporcionar feedback auditivo al usuario
- **Control de Audio**: Automático (siempre reproduce) o inteligente (controlado por la IA según contexto conversacional)
- **Síntesis Previa Forzada**: Garantiza que el agente termine de hablar completamente antes de ejecutar la herramienta
