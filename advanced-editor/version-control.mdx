The Advanced Editor features a powerful integrated Git-based version control system that enables teams to collaborate on AI agents safely and efficiently. The system tracks changes, manages conflicts, and ensures data integrity across multiple users - functioning like a code repository but specifically designed for AI agent development.

## Version Control Interface

The version control interface is located in the top-right section of the Advanced Editor toolbar, displaying real-time collaboration status and version information.

<Steps>
  <Step title="Git Status Indicator">
    Shows the current repository state with color-coded badges:
    - **Clean** (Green): No local changes, synchronized with remote
    - **Modified** (Yellow): Local changes detected with count
    - **Conflicts** (Red): Merge conflicts requiring resolution
    - **Syncing** (Blue): Currently synchronizing with remote
    - **Locked** (Orange): Another user is actively editing
  </Step>

  <Step title="Version Information">
    Displays current local version number in monospace format (e.g., `v42`)
  </Step>

  <Step title="Control Buttons">
    Three primary actions: **Check**, **Pull**, and **Commit**
  </Step>
</Steps>

## Git Status States

The system maintains five distinct collaboration states, each with specific behaviors and available actions.

### Clean State
```
Status: Clean
Version: v42
```
- No local modifications detected
- Fully synchronized with remote repository
- All version control actions available
- Safe to begin editing or switch contexts

### Modified State
```
Status: Modified (3)
Version: v42
```
- Local changes detected (parentheses show number of modified fields)
- Changes tracked in real-time across all configuration sections
- Can commit changes or pull remote updates
- May result in conflicts if remote has newer changes

<Warning>
Modified state indicates unsaved local changes. The system tracks modifications to system prompts, input parameters, output settings, evaluations, and agent configuration automatically.
</Warning>

### Conflict State
```
Status: Conflicts (2)
Version: v42
```
- Merge conflicts detected during pull operation
- Specific fields requiring manual resolution highlighted
- Commit button disabled until all conflicts resolved
- Conflict resolution interface available in Changes tab

### Syncing State
```
Status: Syncing...
Version: v42
```
- Currently communicating with remote repository
- All version control actions temporarily disabled
- Automatic safety timeout after 30 seconds
- Progress indicators show sync status

### Locked State
```
Status: Locked by 2 user(s)
Version: v42
```
- Another collaborator is actively editing the agent
- Pull operations disabled to prevent editing conflicts
- Check operations remain available for version monitoring
- Real-time collaboration status updates

## Version Control Actions

### Check for Updates

The **Check** button queries the remote repository for newer versions without downloading changes.

<Tabs>
  <Tab title="No Updates Available">
    ```
    Local:  v42
    Remote: v42
    Status: Up to date
    ```
    No action required - versions synchronized
  </Tab>

  <Tab title="Updates Available">
    ```
    Local:  v42  
    Remote: v45
    Status: 3 versions behind
    ```
    Pull button enabled to download updates
  </Tab>
</Tabs>

**Check Operation Benefits:**
- Non-destructive query operation
- Available during active editing sessions
- Provides version comparison without changes
- Updates collaboration status indicators

<Info>
Check operations run automatically every 30 seconds in the background to maintain real-time collaboration awareness.
</Info>

### Pull Changes

The **Pull** button downloads and merges remote changes with your local version, following a sophisticated conflict detection algorithm.

<Steps>
  <Step title="Pre-Pull Validation">
    System validates local state and checks for potential conflicts before initiating merge
  </Step>

  <Step title="Change Detection">
    Compares field-by-field differences between local and remote versions
  </Step>

  <Step title="Automatic Merging">
    Non-conflicting changes merge automatically (e.g., different configuration sections)
  </Step>

  <Step title="Conflict Identification">
    Same-field modifications flagged for manual resolution
  </Step>
</Steps>

**Pull Results:**

<Tabs>
  <Tab title="Successful Merge">
    ```
    Before: v42 (Modified - 2)
    After:  v45 (Clean)
    
    Changes Applied:
    - Remote: Updated voice settings
    - Local: Modified system prompt  
    - Result: Both changes preserved
    ```
  </Tab>

  <Tab title="Conflict Detection">
    ```
    Before: v42 (Modified - 1)
    After:  v45 (Conflicts - 1)
    
    Conflict in system_prompt:
    Local:  "You are a sales assistant..."
    Remote: "You are a customer service..."
    ```
    Manual resolution required before proceeding
  </Tab>
</Tabs>

### Commit Changes

The **Commit** button saves your local changes to the remote repository with automatic validation and deployment options.

```typescript
// Commit validation workflow
commitWorkflow(): {
  // 1. Validate agent configuration integrity
  // 2. Check for unresolved conflicts
  // 3. Create commit with descriptive message
  // 4. Optional: Deploy to production environment
  // 5. Update version numbers and sync status
}
```

**Commit Modal Interface:**
- **Commit Message**: Descriptive text for change history
- **Deploy to Production**: Toggle for immediate live deployment
- **Validation Status**: Real-time configuration integrity checks
- **Affected Fields**: Summary of modified sections

<Warning>
Commits with "Deploy to Production" enabled immediately update the live agent. Test thoroughly before enabling production deployment.
</Warning>

## Best Practices for Team Collaboration

### Before Starting Work

<Steps>
  <Step title="Sync Check">
    Always run **Check** to verify you have the latest version before editing
  </Step>

  <Step title="Pull Updates">
    Use **Pull** to download any available changes from teammates
  </Step>

  <Step title="Status Verification">
    Ensure status shows **Clean** before beginning modifications
  </Step>
</Steps>

### During Development

- **Monitor Status Indicator**: Watch for collaboration conflicts in real-time
- **Commit Frequently**: Smaller, focused commits reduce merge complexity
- **Descriptive Messages**: Use clear commit messages for team communication
- **Test Before Commit**: Validate changes in Playground before saving

### Handling Complex Changes

**For Large Modifications:**
1. **Coordinate with Team**: Communicate major changes before starting
2. **Work in Sections**: Break large changes into focused areas
3. **Frequent Commits**: Save progress regularly to minimize conflicts
4. **Integration Testing**: Validate combined changes from multiple contributors

**For Urgent Fixes:**
1. **Priority Communication**: Alert team to urgent changes in progress
2. **Fast Iteration**: Make minimal necessary changes quickly
3. **Immediate Testing**: Validate fix functionality before committing
4. **Production Deployment**: Use automatic deployment for critical fixes

## Troubleshooting Common Issues

### Stuck in Syncing State

If the status remains "Syncing..." beyond 30 seconds:

<Steps>
  <Step title="Wait for Timeout">
    System automatically recovers after 30-second safety timeout
  </Step>

  <Step title="Check Network">
    Verify internet connectivity and API service status
  </Step>

  <Step title="Refresh Editor">
    Reload the Advanced Editor page to reset connection
  </Step>

  <Step title="Contact Support">
    If issue persists, provide version numbers and error details
  </Step>
</Steps>

### Lost Changes During Pull

If local changes disappear after pulling remote updates:

1. **Check Conflict Resolution**: Review Changes tab for merge results
2. **Browser Console**: Look for JavaScript errors during merge process
3. **Version History**: Verify changes weren't automatically merged
4. **Recovery Options**: Contact support with before/after version numbers

<Warning>
The system maintains automatic backups during pull operations, but complex merges may require manual recovery assistance.
</Warning>

### Version Number Misalignment

When local and remote versions show unexpected differences:

```typescript
// Diagnostic information to collect
{
  localVersion: 42,
  remoteVersion: 45,  
  status: "Modified (3)",
  lastSync: "2024-01-15T14:30:00Z",
  pendingChanges: ["system_prompt", "input_params", "evaluations"]
}
```

**Resolution Steps:**
1. Document current state with version numbers
2. Save local changes as backup (copy/paste)
3. Perform fresh Pull operation to resynchronize
4. Manually reapply any lost modifications
5. Commit restored changes with appropriate message

## Security and Permissions

### Access Control

The version control system respects team permissions:

- **Read Access**: View agent configurations and version history
- **Write Access**: Create commits and modify configurations  
- **Admin Access**: Manage collaboration settings and force-resolve conflicts
- **Production Deploy**: Special permission for automatic deployment toggle

### Audit Trail

Every version control operation creates audit records:

- **Commit History**: Complete log of changes with authors and timestamps
- **Conflict Resolutions**: Records of manual merge decisions
- **Deployment Events**: Production deployment logs with rollback capability
- **Access Logs**: User actions and permission usage tracking

This comprehensive version control system ensures that your AI agent development maintains the same professional standards as software development, enabling teams to collaborate confidently while maintaining agent quality and reliability.
